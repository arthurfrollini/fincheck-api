# Raise Combined Coverage to 95%+ Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Raise combined coverage (unit `npm run test:cov` + e2e `npm run test:e2e`, taken together — a line counts as covered if either suite exercises it) from ~75% (unit) / ~75% (e2e) to 95%+ statements across `src/`, by closing the specific gaps identified by diffing both coverage reports against real source.

**Context:** The E2E suite (`2026-07-14-e2e-tests.md`, already merged) intentionally mocks `MailService`, `StorageService`, `BillingService`, `BillingWebhookHandler` and never exercises admin-only routes, the Google OAuth callback, or the Stripe webhook endpoint. Those are exactly the files dragging combined coverage down. This plan closes each gap with the right kind of test — e2e where a real HTTP+DB round trip is the right tool, unit where the code can't be reached through HTTP (OAuth handshake, cron triggers) or exists purely to be mocked out (mail/webhook signature crypto).

**Tech stack:** same as the e2e plan — NestJS `@nestjs/testing`, `supertest`, `ts-jest`, Jest 30, real `fincheck_test` Postgres for e2e; Jest mocks for unit.

## Global Constraints

- Never mock `PrismaService` in e2e specs (unchanged from the original plan)
- Unit spec files: `*.spec.ts` next to the file under test, run by `npm run test:cov` (NOT `test:e2e`)
- E2E spec files: `test/e2e/*.e2e-spec.ts`, extending existing files where the module already has one
- After every task, run **both** `npm run test:cov` and the affected slice of `npm run test:e2e` — a task is not done until both are green
- Do not call real Stripe, Resend, AWS, or Google APIs anywhere. Stripe webhook signature tests use `stripe.webhooks.generateTestHeaderString()` — pure local HMAC, no network call, safe to use with the fake `STRIPE_WEBHOOK_SECRET=whsec_fake` already in `.env.test`
- `Role` enum values: `ADMINISTRATOR`, `USER` (`prisma/schema.prisma`). New users default to `USER`; promoting to admin in a test requires updating the row directly via `app.get(PrismaService)` (same pattern already used in `users.e2e-spec.ts`'s confirm-email test), then re-running `signUpAndGetTokens`'s signin step (or a raw `/auth/signin` call) to get a fresh JWT carrying the new role — the JWT bakes in `role` at signin time (`auth.guard.ts:37-42`), it is not re-read from the DB per request

```ts
import * as request from 'supertest';
import { INestApplication } from '@nestjs/common';
import { createApp } from '../helpers/create-app';
import { cleanDatabase } from '../helpers/db-cleaner';
import { signUpAndGetTokens, uniqueEmail } from '../helpers/auth.helper';
import { PrismaService } from '../../src/shared/database/prisma.service';

describe('Admin routes (e2e)', () => {
  let app: INestApplication;

  beforeAll(async () => {
    app = await createApp();
  });

  afterAll(async () => {
    await app.close();
  });

  afterEach(async () => {
    await cleanDatabase(app);
  });

  async function signUpAdmin(email = uniqueEmail('admin')) {
    const password = 'Test@1234';
    const { accessToken: userToken } = await signUpAndGetTokens(
      app,
      email,
      password,
    );
    const prisma = app.get(PrismaService);
    const user = await prisma.user.findUniqueOrThrow({ where: { email } });
    await prisma.user.update({
      where: { id: user.id },
      data: { role: 'ADMINISTRATOR' },
    });
    // re-signin to get a JWT with the updated role — signup already happened
    const signinRes = await request(app.getHttpServer())
      .post('/auth/signin')
      .send({ email, password });
    return {
      accessToken: signinRes.body.accessToken as string,
      userId: user.id,
      staleUserToken: userToken,
    };
  }

  // Tests below
});
```

```ts
import Stripe from 'stripe';

// inside the describe block, after existing tests:
describe('POST /billing/webhook', () => {
  const stripe = new Stripe('sk_test_fake');
  const secret = 'whsec_fake'; // matches .env.test STRIPE_WEBHOOK_SECRET

  function signedPayload(event: Record<string, unknown>) {
    const payload = JSON.stringify(event);
    const header = stripe.webhooks.generateTestHeaderString({
      payload,
      secret,
    });
    return { payload, header };
  }

  it('returns 401 without stripe-signature header', async () => {
    const res = await request(app.getHttpServer())
      .post('/billing/webhook')
      .send({ type: 'test' });
    expect(res.status).toBe(401);
  });

  it('returns 401 with an invalid signature', async () => {
    const res = await request(app.getHttpServer())
      .post('/billing/webhook')
      .set('stripe-signature', 'not-a-real-signature')
      .send({ type: 'test' });
    expect(res.status).toBe(401);
  });

  it('returns 200 and calls the webhook handler with a valid signature', async () => {
    const event = {
      id: 'evt_test',
      object: 'event',
      type: 'customer.subscription.deleted',
      data: { object: {} },
    };
    const { payload, header } = signedPayload(event);

    const res = await request(app.getHttpServer())
      .post('/billing/webhook')
      .set('stripe-signature', header)
      .set('Content-Type', 'application/json')
      .send(payload);

    expect(res.status).toBe(200);
    expect(res.body).toEqual({ received: true });
    expect(mockBillingWebhookHandler.handle).toHaveBeenCalledTimes(1);
  });
});
```

  Note: supertest's `.send(payload)` with a JSON *string* body and an explicit `Content-Type: application/json` header sends the raw string as-is (does not re-serialize) — this must exactly match the string used to compute `header`, since Stripe signature verification hashes the exact raw bytes. If this exact combination doesn't produce a byte-identical raw body server-side (check `req.rawBody` — enabled repo-wide by the `rawBody: true` fix already applied to both `main.ts` and `test/helpers/create-app.ts`), adjust by sending a `Buffer.from(payload)` instead of the string, whichever supertest/Nest combination preserves raw bytes untouched. Verify by running the test — don't guess blindly, check the actual `res.status`.
